# CFS-Suite v6.5 Release Notes

> 发布日期：2026-06-23
> 主线版本：`v6.5.0`
> 上一版本：`v6.4.0`（砍 PETL + 静默清 [CFS4_AUTO] 残留标签）

---

## 概览

v6.5 在 v6.4 砍 PETL 之后回归"CFS-Suite 的优化边界在哪里"这个问题，给出三方向答案：

1. **PSIS R1 自动化补强**（P3）—— 消除"二次切卡 db/mvu 不重扫"的用户体感延迟
2. **SEM 还原 4.9 主动迁移**（P1）—— 把纯静态长 entry 推 prefix 区参与 cache
3. **RSI Drift Panel 一键修复**（P2）—— 把"识别了不能动"升级为"用户授权点击就修"

> P3 + P1 已在本文档说明。P2 在后续 commit 落地后追加。

---

## 自动后台链路 vs 用户授权链路（架构边界声明）

CFS-Suite 内部对 worldbook entry 的所有写操作分两条链路：

### 自动后台链路（CFS 自己跑，不弹 confirm）

- **PSIS R1**：扫描三大块（数据库 / MVU / 动态注入）的 position 漂移，自动归零到 `at_depth_as_user/depth=0`
- **kernel.js audit**：CFS 自管 `[CFS4_*]` entry 漂移自动修复
- **cfs4_auto_cleanup.js**：启动期一次性清掉 PETL 残留的 `[CFS4_AUTO]` 标签

**为什么这些可以自动？** 它们修的是 "position 漂移"——把 entry 从错误位置恢复到正确位置，不改变用户的 cache 命中物理结构，只是恢复 CFS 设计的预期状态。

### 用户授权链路（必须点击触发）

- **SEM 主动迁移**：把纯静态长 entry 推 `before_character_definition`（prefix 区）
- **RSI Drift Panel 一键修复**（v6.5 新增）：改外部注入型 entry 的 position 或追加 `[cfs:ignore]` 标记
- **MVU 控制台一键操作**：启用/禁用/修回 MVU 接口

**为什么这些必须用户决定？** 它们改变 cache 命中物理结构（把 entry 搬进/搬出 prefix 区），是用户对 cache 优化策略的决策，CFS 不替用户拍板。

## P3 改动详情

### 1. 二次切卡全量重扫三大块（psis.js）

**问题**：v6.4 之前，`chat_id_changed` 事件触发后只跑 `applyFixesByCategory('dynamic', ...)`，db 类和 mvu 类不重修。用户切到新卡 / 切回旧卡时，db/mvu 类的 position 漂移不会自动修。

**v6.5**：改为 `applyFixesByCategory('all', ...)`——切卡后 1.5s 自动重扫修复三大块。

**保持兼容**：LS toggle key `cfs-suite/auto_zero_dynamic_on_chat_change` 名字不变（用户原有配置不破坏），但行为已扩展到三大块。

**性能**：典型卡 db+mvu+dynamic 合计 5-15 个 entry，每 patch ~200ms，1.5s debounce 内完成。

### 2. 胶囊状态条加 PSIS R1 指示符（floating_capsule.js）

**问题**：用户进卡后默认不会点开胶囊面板，看不到 PSIS R1 "扫描了多少条 / 修复了多少条"的反馈，体感是"没自动"。

**v6.5**：胶囊文字后追加 PSIS R1 状态指示符：

| 状态 | 显示 |
|---|---|
| 启动期 PSIS R1 OK + 修复 N 条 (N>0) | `CFS缓存优化器 · 接管已启用 · 🛡修复N` |
| 启动期 PSIS R1 OK + 无违规 | `CFS缓存优化器 · 接管已启用 · 🛡✓` |
| 启动期 PSIS R1 失败 | `CFS缓存优化器 · 接管已启用 · 🛡✗` |
| 切卡 PSIS R1 跑过 | 文字同步更新 |
| PSIS R1 未上报 | 不显示后缀（向前兼容） |

**实现**：从 `NotificationCenter._peekPending().psis` 读最近 report，每 2s 轮询时同步。鼠标 hover 胶囊文字显示完整 summary tooltip。

**为什么不直接弹 toast 提示**：v6.4 已经在启动期弹 `flushStartup` toast 包含 PSIS 启动报告，但 toast 一闪而过用户可能错过。胶囊文字是**长期可见**的指示，进入卡片任何时候看胶囊都知道 PSIS R1 自动跑过。

### 3. 延迟链路说明（README + 本文档）

PSIS R1 启动期自动扫描有 ~7s 左右延迟，链路是：
1. 宏 cache 预填 2s + 5s 异步加载（`kernel.js:837-839`）
2. SessionGate 异步等就绪
3. Coordinator plugin `onSessionReady` 异步触发
4. PSIS R1 `psisScanR1Violations + psisApplyR1Fix`（`psis.js:1817-1842`）

如胶囊状态条 `🛡` 后缀显示 "✓" 或 "修复N"，即代表三大块 depth 已归零 —— 不需手动打开胶囊触发。

## P1 改动详情

### 核心定位：SEM 与 WM 不同向

v6.3 / v6.4 期间 SEM 因为 PETL 误识别引入的反优化（"密度 < 5/1000 字符" 判定漏掉单个动态宏即击穿 cache）被降级为"仅展示候选不主动迁，让位 WM"。

v6.4 砍 PETL 后这套分工失效——含动态宏的 entry 没有 CFS 工具管（只能等 WM 用户手动），而 SEM 自己被降级到"灰色不推荐"。这是逻辑矛盾。

**v6.5 还原 4.9 主动迁移**：

- SEM 管：`PSIS pattern getEntryRiskLevel() === 'safe'` 的纯静态长 entry → `before_character_definition + constant=true`（prefix 区，参与 cache 缓存）
- WM 管：含动态宏（`{{lastusermessage}}`、`getvar()`、EJS、骰子等）的 entry → `at_depth_as_user/depth=0`（chat 末尾）
- 两者管的**区间物理隔离**：SEM 推 prefix 区的 entry 是 PSIS 判 safe 的，不可能命中 WM 的动态宏判定。不会互殴。

### 改动清单（cfs/modules/sem.js）

| 行号 | 改动 |
|---|---|
| L516-517 | hint 从 "推荐用 WM...SEM 仅兜底" → "扫描纯静态长 entry → 用户授权迁到 prefix 区参与 cache。含动态宏的 entry 不进候选（PSIS pattern 精准识别）" |
| L530-534 | drift 提示从 "多数情况下这是正确的" → "已迁移条目当前不在 prefix 区（被外部改回）。如确认是纯静态长内容，到「📋 已迁移列表」一键重迁回 prefix"，颜色 `#888` → `#ffa726` |
| L595-599 | drift 警告去掉"PETL 自动..."（PETL 已砍）→ "SEM 候选都是纯静态长 entry（不含动态宏），漂移多是外部工具改回。可重迁回 prefix 恢复" |
| L604 | 按钮 `cfs-btn` `style="opacity:0.7"` → `cfs-btn cfs-btn-primary`，文案 "强制重迁回 prefix" → "重迁回 prefix"（去"强制"中性化） |

### 保留项（v6.5 不动）

- `_semScanCandidates` PSIS pattern 精准识别（`sem.js:141-200` `_semEntryIsRisky` 调 `PSISPatterns.getEntryRiskLevel()`）—— 6/22 下半场已修好，绝不能滚回到宏密度判定
- 跨卡污染修复 `activeOnly=true`（`sem.js:364-388, 476`）
- `chat_id_changed` 监听 800ms debounce（`sem.js:901-924`）
- Audit Guard `_semAuditDrift`（`sem.js:423-448`）
- `SEM_CONFIG.MIN_LEN=500` / `PRECHECK_LEN=1000` 阈值不变
- LS key `cfs_sem_migrations_v1` 不变（v6.3/v6.4 迁移记录可读）
- `window.CFS4.SEM` API 暴露面（`sem.js:880-887`）不变

### 话术红线（v6.5 必须遵守）

- **禁用"实测命中率 +X%"类话术**：4.9 时期的命中率数字（84.8% 等）没有 v6.5 真机数据兜底，照搬等于欺诈
- **改用物理事实表述**：`prefix 区参与 cache`、`PSIS pattern 判 safe`、`不含动态宏` —— 这些是字节级物理事实，不依赖任何实测数据

### 用户授权链路（架构纯净度）

SEM 还原后所有写操作仍走"用户点击 → 跑 `_semMigrate`/`_semRollback` → 1500ms 回读验证"三段式。`_semPassiveScan` 仅做候选数 toast 提醒（`TOAST_THRESHOLD=3`），不写。

**没有任何自动迁移**。这条边界 4.9 已经画好，v6.5 不破。

## 不在 v6.5 范围（避开复杂度）

- **PSIS R1 触发时机挪到宏 cache 完成后**：需要 kernel.js 暴露 `onMacroCacheReady` 事件、PSIS 改订阅模式。性能优化路径不是正确性路径，留 v6.6+
- **预设切换 / prompt_order 重构**：不是 CFS-Suite 的职责（详见 v6.4 评估报告 §四）
- **复活 PETL**：永久排除（v6.4 PETL 移除日志已说明原罪）

## 验证步骤

### 启动期 PSIS R1 自动跑

1. F12 console 启动前打开 → 进卡
2. 看 console 输出 `[PSIS R1] 启动扫描: 发现 N 条违规` + `[PSIS R1] 修复 M / N 条`
3. 进卡后 ~7-10s 内，胶囊文字后缀出现 `🛡修复M` 或 `🛡✓`
4. hover 胶囊看 tooltip 显示完整 summary

### 二次切卡全量重扫

1. A 卡进 → 切到 B 卡
2. F12 看 `[CFS-PSIS] ⚡ 切卡自动归零: N 条 (db/mvu/dynamic 三大块)`
3. N 应含三类总和（不再仅是 dynamic）
4. 胶囊文字后缀同步更新到 `🛡修复N`

### LS toggle 关闭

```js
localStorage.setItem('cfs-suite/auto_zero_dynamic_on_chat_change', '0');
```

切卡不再自动跑（兼容旧 toggle）。
